ANALYSIS

NEED OF HASHING

The amount of data on the internet is growing exponentially every day, making it difficult to store it all effectively. In day-to-day programming, this amount of data might not be that big, but still, it needs to be stored, accessed, and processed easily and efficiently.

Arrays is one of the way but not very efficient. Though storing in Array takes O(1) time, searching in it takes at least O(log n) time. This time appears to be small, but for a large data set, it can cause a lot of problems and this, in turn, makes the Array data structure inefficient.

Hash tables offer several advantages over arrays, particularly in scenarios that involve frequent searching, inserting, or deleting operations. Here's a brief comparison of the advantages of hash tables over arrays:

1. Faster Operations: Hash tables offer O(1) average time for search, insert, and delete, compared to O(n) in arrays.

2. Dynamic Sizing: Hash tables can dynamically resize, while arrays have fixed sizes (unless reallocated).

3. Dynamic Sizing: Hash tables can dynamically resize, while arrays have fixed sizes (unless reallocated).

4. No Need for Sorted Data: Hash tables don't require data to be sorted for fast lookup, unlike sorted arrays where binary search can be used.

TIME COMPLEXITY

The time complexity of hashing depends on the operations and the efficiency of the hash function used.

1. Average Time Complexity

  • Search: O(1)
  • Insert: O(1)
  • Delete: O(1)

2. Worst Time Complexity

  • Search: O(n)
  • Insert: O(n)
  • Delete: O(n)

SPACE COMPLEXITY

The space complexity of a hash table mainly depends on the number of elements (keys) and the size of the table.

1. Average Space Complexity

  • O(n), where n is the number of elements (keys) stored in the hash table.

2. Worst Space Complexity

  • O(n + m), where n is the number of elements and m is the size of the hash table (including empty spaces)
